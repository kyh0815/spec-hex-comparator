# 현·신(As-Is/To-Be) 데이터 비교 검증 도구 — 구현 스펙서

> **이 문서의 위치**
> 마스터 세션(기획/방향성)이 작성한 구현 지시서. **Claude Code**가 이 문서를 받아 기능을 구현하고, **Claude Design**이 비주얼을 담당한다.
> 이 문서는 *무엇을/왜/어떻게 검증하는가*를 확정한다. *어떻게 보이는가*(색·폰트·레이아웃·모션)는 의도적으로 비워두었으며 §8을 참고할 것.
> 빈칸을 임의로 채우지 말 것. 스펙에 없는 동작이 필요하면 구현 전에 질문할 것.

---

## 1. 목적 (Why)

메인프레임(As-Is) → 오픈 시스템(To-Be, PostgreSQL 추정) 마이그레이션 검증의 **결과 책정 도구**.

기존에 DB에서 두 테이블을 덤프·비교하는 도구가 따로 존재하나, 출력이 평문 로그라 *어느 row의 어느 값이 다른지*와 *최종 합격 여부*가 한눈에 들어오지 않는다. 이 도구는 그 공백을 메운다.

핵심 전제 (변하지 않음):
- **인코딩 변환은 이미 완료**되어 있다. As-Is/To-Be 양쪽 모두 최종 상태다. 이 도구는 **변환하지 않고 byte-for-byte로 비교만** 한다.
- **추출 정규화(NULL/공백/포맷 통일)도 외부에서 완료**되었다고 가정한다. 이 도구는 정규화하지 않는다.
- 데이터는 **이미 양쪽에 존재**한다. 이 도구는 **생성·추출·트리거를 하지 않고 비교만** 한다.

## 2. 범위 (Scope)

### In scope
- 헤더가 있는 CSV 2개(As-Is, To-Be)를 업로드받아 비교
- PK 기준 row 매칭
- 사용자가 지정한 검사 대상 컬럼에 대해 값의 완전 일치(문자열 동일) 비교
- PASS/FAIL 판정 + FAIL 사유 분기
- 결과 화면 표시 및 판정 리포트 CSV 내보내기

### Out of scope (구현하지 말 것)
- DB 직접 연결/쿼리/덤프 (입력은 오직 CSV 파일)
- 인코딩 변환, 정규화, 트림, 포맷 보정 — **일절 하지 않는다.** 값은 CSV에서 읽은 문자열 그대로 비교한다.
- 데이터 생성/수정/저장
- 서버 전송 (§7 보안 참조)
- row+컬럼 단위 핀포인트 예외 등록(known-diff registry) — 이번 버전 비대상. 컬럼 단위 제외로 충분.

## 3. 입력 명세 (Input)

- **형식**: 헤더 행 + 컬럼 분리형 CSV. 첫 줄이 컬럼명, 이후 각 줄이 데이터 row.
- **파일 수**: 2개 (As-Is, To-Be). 각각 별도 업로드.
- **인코딩**: 임의(이미 인코딩 완료 전제). 파서가 표준적으로 읽는다. (구현: PapaParse `header: true, skipEmptyLines: true`)
- **컬럼명**: As-Is와 To-Be의 컬럼명은 **동일**하다고 가정. (실데이터 예: `UNIQUE_NUMBER, FBWI5M, FBWI6M, FBWI7C … FBWIZD`)
  - 단, 양쪽 컬럼 집합이 완전히 같지 않을 수 있으므로 **공통 컬럼(교집합)만** 비교 후보로 노출한다.
- **값**: 컬럼별 헥스 문자열(예: `4A6F...`)이 일반적이나, 임의 문자열일 수 있다. 비교는 값의 형식과 무관하게 문자열 동일성으로 수행한다.

## 4. 데이터 모델 / 설정 (Configuration)

사용자가 비교 실행 전 지정하는 것:

1. **PK 컬럼 (매칭 키)** — 공통 컬럼 중 1개 선택.
   - 자동 추측: 정규식 우선순위 `^(unique_number|id|pk|key)$` → `(id|number|no)$` → 첫 컬럼. 사용자가 변경 가능.
2. **검사 대상 컬럼 (검사/제외 2-state)**
   - PK를 제외한 모든 공통 컬럼이 **기본값 ON(검사)**.
   - 사용자는 개별 컬럼을 **OFF(제외)** 로 토글 가능. 제외된 컬럼은 비교에서 빠지고 판정에 영향 없음.
   - 최소 1개 이상 검사 대상이어야 실행 가능.
3. **바이트 단위 diff 표시 (on/off, 표시 전용 옵션)**
   - 상세 보기에서 다른 부분을 하이라이트할 때, 값이 헥스로 판별되면(짝수 길이 + 16진수 문자) 2자리(=1바이트)씩 끊어 비교·표시. off면 문자 단위.
   - **주의: 이는 표시 방식일 뿐 판정 로직과 무관.** 판정은 항상 전체 문자열 동일성으로 한다.

## 5. 비교 알고리즘 (Core Logic — 결정론적, AI 미사용)

```
입력: asIsRows[], toBeRows[], pk, inspectCols[] (검사 대상 컬럼 목록)

1. 인덱싱
   - asIs를 pk값 → row 로 Map 구성. toBe도 동일.
   - pk값이 비어있는(null/"") row는 스킵.
   - pk 중복 발생 시: 첫 row만 사용하고, 중복 키 목록을 경고로 수집.

2. 키 합집합 순회 (As-Is 키 ∪ To-Be 키)
   각 key에 대해:
   - As-Is에만 있음        → status = ONLY_AS_IS   (사유: MISSING_IN_TOBE)
   - To-Be에만 있음        → status = ONLY_TO_BE   (사유: EXTRA_IN_TOBE)
   - 양쪽 존재:
       diffCols = inspectCols 중 String(asIs[c]) !== String(toBe[c]) 인 컬럼
       - diffCols 비어있음   → status = MATCH
       - diffCols 있음       → status = MISMATCH (사유: VALUE_MISMATCH, diffCols 기록)

3. 집계
   - total, matched, mismatched, onlyAsIs, onlyToBe
   - dupesAsIs[], dupesToBe[]  (중복 키)
   - inspectedCols[], excludedCols[]  (검사/제외 컬럼 목록)
```

**비교는 `String(a) !== String(b)` 단순 동일성.** trim·대소문자·숫자정규화 등 어떠한 보정도 하지 않는다(전제상 외부에서 완료). 보정을 넣으면 안 된다.

대용량 대비: 비교 자체는 O(n)으로 전체 수행하되, 화면 목록 렌더링은 상한(예 500건)을 두고 "전체는 내보내기 사용" 안내. (정확한 상한은 성능 보고 Claude Code가 조정 가능)

## 6. 판정 모델 (Verdict)

**원칙: PASS는 검사 대상 기준 완전 일치일 때만 부여한다. 보수적으로 판정한다.**

```
PASS  ⟺  mismatched == 0  AND  onlyAsIs == 0  AND  onlyToBe == 0
FAIL  ⟺  그 외 전부
```

- 판정은 **PASS / FAIL 이분법** (단호하게).
- 단, **FAIL 사유는 분기하여 각각 카운트**한다:
  - `VALUE_MISMATCH` — 검사 컬럼 값이 다른 row 수
  - `MISSING_IN_TOBE` — As-Is에만 있는 row 수
  - `EXTRA_IN_TOBE` — To-Be에만 있는 row 수
- **제외 컬럼은 판정에 영향 없음.** 다만 **반드시 결과에 명시**한다 — "전체 N개 중 M개 검사 (제외: col_x, col_y)". *무엇을 검사하지 않았는가*는 검증 결과서의 신뢰성에 직결되는 1급 정보다. 누락하지 말 것.
- PK 중복은 판정을 막지 않지만, 발생 시 결과에 경고로 표시(비교 정확도 영향 가능성).

## 7. 비실행 요구사항 (Non-functional)

- **전 처리 클라이언트 사이드(브라우저 내)**. 데이터는 서버로 전송되지 않는다. 엔터프라이즈 마이그레이션 데이터이므로 이 경계는 필수.
- 단일 배포 단위(웹 컴포넌트/아티팩트). 외부 런타임 의존 최소화.
- CSV 파싱은 PapaParse 사용 권장(따옴표/이스케이프 엣지 케이스 처리).
- **localStorage/sessionStorage 등 브라우저 스토리지 사용 금지** (아티팩트 환경 제약). 상태는 메모리(React state)로만 유지.

## 8. 화면 요구사항 (Information & Priority — *표현은 Claude Design 몫*)

> 아래는 **각 화면에 무엇이 어떤 우선순위로 존재해야 하는가**만 규정한다.
> 색·폰트·여백·레이아웃·모션·아이콘 등 **시각 표현은 일절 지정하지 않는다.** Claude Design이 결정한다.
> 단 하나의 제약: **헥스/원시 데이터 값은 등폭(monospace)으로 표시**할 것(자리 비교 가독성). 그 외 자유.

### 화면 A — 업로드 & 설정
- As-Is, To-Be 두 파일의 업로드 진입점(드래그앤드롭 + 클릭). 어느 쪽이 As-Is/To-Be인지 명확히 구분.
- 업로드 후 각 파일의 인식된 컬럼 목록을 확인할 수 있을 것.
- PK 선택 컨트롤(공통 컬럼 기준, 자동 추측값 표시).
- 검사/제외 컬럼 토글 (기본 전체 검사). 각 컬럼의 현재 상태(검사 중 / 제외됨)가 한눈에 구분되어야 함.
- 바이트 단위 diff 표시 옵션.
- "비교 실행 및 판정" 실행 액션.
- 입력 오류(미업로드, PK 미선택, 검사 컬럼 0개)는 명확한 메시지로 안내.

### 화면 B — 판정 결과 (최상단, 최우선)
- **PASS / FAIL 판정이 화면에서 가장 먼저·가장 강하게 인지**되어야 함. (이 도구의 존재 이유)
- 판정 근거 한 줄 요약: PASS면 "완전 일치"(또는 제외 컬럼 있음 명시), FAIL이면 사유별 건수.
- 검증 메타: PK 컬럼명, 검사 컬럼 수 / 제외 컬럼 수(+제외 목록).

### 화면 C — 집계 (요약 지표)
- 다음 수치가 동시에 보일 것: 전체 / 일치 / 불일치(VALUE_MISMATCH) / As-Is만(MISSING_IN_TOBE) / To-Be만(EXTRA_IN_TOBE).
- PK 중복 경고(있을 때만).

### 화면 D — 상세 목록
- 결과 row 목록. 각 항목: PK값, 상태(일치/불일치/한쪽만), 다른 컬럼명 미리보기.
- 필터: 불일치만 / 일치만 / 전체 (기본은 문제 있는 것 우선 노출).
- 불일치 row를 펼치면 **컬럼별 As-Is 값 vs To-Be 값을 나란히** 보여주고, **다른 부분을 하이라이트**(바이트 또는 문자 단위, 옵션 따라). 빈 값도 명확히 구분 표시.
- 목록은 성능 상한까지만 렌더, 초과 시 안내.

### 화면 E — 내보내기
- 판정 리포트를 CSV로 내보내기.
- 포함 정보: 상단에 메타(verdict, total, mismatched, missing/extra 건수), 이후 row별 (key, status, diff_columns).
- 제외 컬럼 목록도 메타에 포함.

## 9. 엣지 케이스 (반드시 처리)

| 케이스 | 처리 |
|---|---|
| 한쪽 파일만 업로드 | 실행 차단 + 안내 |
| 공통 컬럼 0개 | 실행 불가 안내 (컬럼명 불일치 의심 메시지) |
| PK 컬럼에 빈 값 row | 해당 row 스킵 (인덱싱 단계) |
| PK 중복 | 첫 row 사용 + 경고 수집·표시 |
| 검사 컬럼 전부 제외 | 실행 차단 + 안내 |
| 한쪽에만 존재하는 컬럼 | 공통 컬럼만 비교 후보로(교집합) |
| 값 길이가 양쪽 다름 | diff에서 길이 차이도 시각적으로 드러나게(한쪽에만 잔여 유닛) |
| 매우 큰 파일 | 비교는 전수, 목록 렌더만 상한 |

## 10. 분업 경계 (Roles)

- **마스터 세션(기획)**: 본 스펙. 비교/판정의 의미·규칙 확정. 변경 요청의 최종 판단.
- **Claude Code(구현)**: §3–§7, §9의 로직을 결정론적으로 구현. §8의 정보·우선순위 요구사항 충족. 디자인 표현은 Claude Design 산출물에 맞춤. 스펙 외 동작 임의 추가 금지.
- **Claude Design(비주얼)**: §8의 요구사항을 만족하는 시각 디자인. 단일 monospace 제약 외 자유. 단, "PASS/FAIL이 가장 강하게 인지된다"는 정보 우선순위는 디자인에서도 유지.

## 11. 수용 기준 (Acceptance)

- [ ] As-Is/To-Be CSV 2개 업로드 → 공통 컬럼 자동 인식
- [ ] PK 자동 추측 + 수동 변경 동작
- [ ] 검사/제외 토글이 판정에 정확히 반영 (제외 컬럼 차이는 PASS를 막지 않음)
- [ ] PASS 조건이 "불일치 0 + 누락 0 + 초과 0"으로 정확히 동작
- [ ] FAIL 사유 3종이 각각 정확히 카운트
- [ ] 제외 컬럼 목록이 결과·리포트에 명시됨
- [ ] 상세에서 다른 바이트/문자가 정확히 하이라이트
- [ ] 어떤 보정도 없이 `String` 동일성으로만 비교 (trim/대소문자/숫자정규화 없음)
- [ ] 데이터가 서버로 나가지 않음 (클라이언트 전용)
- [ ] 판정 리포트 CSV 내보내기 동작
